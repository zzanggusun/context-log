## 1. 한 줄 정의

사용자가 금융 지식·전략에 대한 **의도를 자연어로 표현하면**, AI와 검증된 시스템 기반 위에서 그것을 **분석·검증·구현·체험**할 수 있게 해주는 금융 리터러시 플랫폼.

비유: **금융계의 깃허브 + 트레이딩뷰**. 대형 서비스는 정제된 완성품을 제공하지만 개개인이 자기 상황·전략·지식을 시스템 위에서 직접 굴려볼 수는 없고, 개인이 AI로 직접 만들자니 시스템화의 벽에 막힌다. 우리는 그 사이에서 **검증된 기반 위의 자유도** — 안전하게 실험하고, 가져온 정보를 판정받는 자유 — 를 제공하는 **금융 민주화 플랫폼**이다.

## 2. 문제 정의 (Problem)

### 2.1 타겟이 겪는 문제

| 타겟 | 문제 |
|------|------|
| 20대 주식 입문 고려층 | 시작하고 싶지만 뭘 모르는지도 모름. 진입 장벽이 지식이 아니라 "지식을 다룰 수단" |
| 전연령 기존 투자자 중 감(직관) 기반 투자층 | 수식/지표 적용 방법을 모르고, 자기 포트폴리오의 리스크가 뭔지 몰라 막연히 두려워하며 투자 |

**핵심 페르소나 — 탐색형 학습자**: 경제 공부를 마치고 시작하는 것이 아니라, 일단 사보고 → 차트를 들여다보고 → 기술적 분석·금융 이론·유튜브의 다양한 관점을 부딪혀가며 쌓아가는 사람. 이들에게 자유도는 부담이 아니라 이미 하고 있는 행동이며, 플랫폼의 역할은 그 실험을 검증된 안전한 환경으로 옮겨주는 것이다.

### 2.2 두려움의 근원 (핵심 진단)

두려움의 원인은 지식 부족 자체가 아니라 다음 세 가지다.

1. **지식을 자기 상황에 적용할 수단이 없음** — 책·유튜브·커뮤니티에서 본 전략을 실제로 구현하고 굴려볼 방법이 없음
2. **접한 정보를 믿어도 되는지 판단할 능력이 없음** — 정보가 수학적·사실적으로 맞는지, 알려진 약점과 적용 조건이 무엇인지 판단하기 어려움
3. **아이디어를 시스템으로 구현할 기술적 경로가 없음** — 바이브 코딩과 코딩 에이전트가 발전해도 무엇부터 시작해야 하는지, 금융 아이디어를 어떤 데이터·규칙·상태·체결 조건으로 변환해야 하는지, 생성된 결과를 어떻게 검증해야 하는지 모르는 사용자가 많음

코드 생성 비용은 낮아졌지만 금융 의도를 안전하고 검증 가능한 소프트웨어로 만드는 시스템화의 장벽은 여전히 남아 있다.

플랫폼은 이 문제를 **적용 수단 제공 + 판단력 대여 + 구현 경로 제공**으로 해결한다. 사용자는 개발자가 되지 않아도 자신의 금융 지식과 전략을 구현·관찰·검증할 수 있다.

판단력 대여는 최종 목적이 아니라 시작점이다. 사용자는 처음에는 플랫폼의 검증 능력을 빌리지만, 반복적으로 출처·가정·실행 결과·반례를 비교하면서 금융 정보를 비판적으로 받아들이는 습관을 형성한다.

따라서 플랫폼의 금융 리터러시는 지식을 일방적으로 전달하는 교육이 아니다. 사용자가 정보를 직접 적용하고, 결과를 관찰하고, 근거와 한계를 판단하는 과정을 반복하게 만드는 **체험형 판단력 훈련**이다.

### 2.3 "그냥 ChatGPT에 물어보면 되잖아"가 안 되는 이유 (= 사업 근거)

개인이 AI에게 물어서 코드 조각·설명은 얻을 수 있다. 그러나:

- 그것을 검증된 데이터, 주문 실행, 리스크 관리, 백테스트 환경이 갖춰진 **시스템 위에서 안전하게 동작하게 만드는 것**은 완전히 다른 문제
- 코딩 에이전트가 있어도 대부분의 개인이 이를 처음부터 안전하게 시스템화하고 지속적으로 검증·유지하는 데는 높은 비용이 듦
- 이 "시스템화의 어려움"이 플랫폼의 존재 이유

**AI 신뢰의 모순 (신뢰 → 검증 수요)**

"사용자들이 AI 기반 금융 서비스를 신뢰하겠는가?"라는 우려는 그 자체로 모순이다. 이미 사람들은 AI를 자신의 업무 — 성과와 책임이 걸린 영역 — 에까지 일상적으로 활용할 만큼 높은 신뢰를 형성했고, 그 신뢰는 수년간의 AI 발전과 실제 성과가 뒷받침한다. 신뢰의 진입 장벽은 사실상 이미 허물어졌다.

진짜 문제는 방향이 반대다: 신뢰가 부족해서가 아니라 신뢰가 충분히 높기 때문에, 범용 AI가 정량 금융에서 확신에 차서 틀릴 때(수치 오류, 실시간 데이터 부재, 백테스트하면 무너지는 논리) 사용자는 그것을 걸러낼 수 없다. 유튜브가 하던 역할을 AI가 더 개인화된 형태로 승계한 것이다.

따라서 구조는 역할 분담이다 — **신뢰는 사용자의 AI가 문을 열어주고, 검증 레이어는 그 신뢰가 배신당하지 않게 한다.** 높은 AI 신뢰는 진입 마찰을 없애는 유통 자산이고, 그 신뢰에 접지(grounding)를 제공하는 검증이 플랫폼의 가치다. 사람들의 AI 신뢰가 높아질수록 이 플랫폼의 필요는 커진다.

### 2.4 Why Now — 왜 지금인가

- **돈의 개념 전환**: 전세계적 화폐 가치 하락으로 "돈을 보유하는 것"에서 "자산에 위치시키는 것"으로 개념이 이동 중. 투자는 선택이 아니라 자본주의에서의 생존 요건이 되어가고 있다.
- **AI·기계의 노동 대체**: 노동 소득의 비중이 구조적으로 줄어들며, 개인은 금융시장을 통해 핵심 섹터(반도체 등)에 자본을 위치시켜 수익을 얻고, 그 자본이 국가 기반 산업 발전으로 순환하는 구조가 강화된다.
- **실측 신호**: 한국에서 부동산 자금의 증시 유입, 주식 인구의 뚜렷한 증가 — 전세계적 현상의 로컬 확인. (실측 데이터 보강 필요 — 7.3 참조)

**함의**: 투자가 "의무"가 되는 시대에 유입 인구의 다수는 준비 없이 떠밀려 들어온다. 금융 리터러시 인프라의 수요는 구조적으로, 비가역적으로 커진다.

## 3. 핵심 자산 & 해자 (Moat)

### 3.1 축적된 방법론 = 핵심 IP

`open-trading-api` 레포는 상품이 아니라 **증명 케이스**:

- 비전공자가 AI와 1년+ 협업으로 유지보수 가능한 복잡한 금융 시스템을 실제로 구축할 수 있음을 증명
- 그 과정에서 축적된 것이 진짜 자산:
  - 작업 흐름(워크플로우) 패턴
  - 컨텍스트 문서 (CLAUDE.md, MEMORY.md 등)
  - ADR 체계와 규율 (불변, supersede, 구현 전 작성)
  - 커밋 규율 (구조 변경/동작 변경 분리)
  - 리팩토링 전략 (위임 패턴, 점진 분리, 리스크 레벨별 커밋 분할)
  - "AI가 어디서 실패하고 무엇을 강제해야 하는지"에 대한 노하우

**플랫폼의 핵심 IP = "AI에게 금융 시스템을 안전하게 만들게 하는 방법론" 그 자체.**

### 3.2 프랙탈 구조 (개인 워크플로우 → 제품 구조)

```
[Lee] ↔ [Claude Code] ↔ [레포 + 컨텍스트 문서(CLAUDE.md/ADR)]
                ↓ 스케일업
[사용자] ↔ [사용자의 AI] ↔ [플랫폼 + 플랫폼 제공 아키텍처 컨텍스트]
```

Lee가 Claude Code에 컨텍스트를 먹여 협업했듯, 플랫폼이 모든 사용자의 AI에게 컨텍스트를 제공한다. 축적된 ADR/규칙 문서가 3단계에서 외부 AI에 제공할 컨텍스트의 원형.

### 3.3 검증 가능성의 확장 — 전략 성과에서 지식 계보까지

기존 퀀트·백테스트 플랫폼도 수익률, MDD, 샤프 등 전략 성과 측정을 제공한다. 플랫폼의 차별점은 이러한 지표 자체가 아니라 자연어·영상·문서 등 비정형 금융 지식을 실행 가능한 명세와 검증 과정으로 변환하는 데 있다.

검증 대상은 백테스트 전략에 한정되지 않는다. 기술적 분석, 기업 분석, 재무 이론, CFA 지식, 거시경제와 지정학적 주장 등 다양한 금융 지식을 사실적·논리적·경험적·시간적 관점에서 검토할 수 있다.

플랫폼은 전략과 금융 지식을 다음의 독립된 검증 축으로 평가한다.

1. **의도 일치도**: 사용자 원문과 해석된 명세가 일치하는가
2. **구현 검증도**: 생성된 로직이 명세대로 동작하는가
3. **사실·논리 검증도**: 주장과 근거 사이에 오류나 모순이 없는가
4. **통계적 유효성**: 데이터 누수·오버피팅·생존편향 등을 통제했는가
5. **출처 추적성**: 원본 정보와 추출·해석 근거를 확인할 수 있는가
6. **재현 가능성**: 다른 데이터·기간·사용자가 결과를 재현할 수 있는가

사용자는 현재의 지식으로 과거의 판단을 다시 검토하고, 과거와 현재의 지식이 어떻게 연결·충돌·발전했는지 확인할 수 있다. 플랫폼은 출처, 당시 가정, 이후의 반례와 수정 이력을 보존하여 개인의 금융 지식이 변화하는 계보를 만든다.

프런티어 모델과 축적된 지식은 교차 검증자이자 관계 탐색기로 활용한다. 그러나 모델의 판단 자체를 정답으로 취급하지 않으며 출처·시점·반대 근거·불확실성을 함께 제시한다.

백테스트 성과는 강력한 증거지만 전략 품질을 자동 확정하는 유일한 라벨은 아니다. 요청·해석·구현·검증·수정·승인의 전체 계보가 전략과 지식 자산의 신뢰도를 형성한다.

### 3.4 Founder-Market Fit

창업자 자신이 금융·소프트웨어 비전공자로서 이 탐색 경로를 6년간 직접 걸어왔음. 타겟의 니즈·막히는 지점·학습 순서에 대한 이해가 관찰이 아닌 경험에서 나옴.

### 3.5 도메인 컨텍스트 플라이휠 (Human Leverage)

플랫폼의 또 하나의 핵심 자산은 **사용자가 외부에서 가져오는 금융 컨텍스트**다. 사용자는 유튜브·책·커뮤니티·블로그·리포트·외부 AI 대화 등에서 각자 시간과 비용을 들여 지식과 의도를 만든다. 그러나 그것을 실제 전략·워크플로에 적용하려면 결국 플랫폼 안으로 가져와야 한다. 플랫폼은 이를 단순 저장하지 않고 전략 가정·필요 데이터·리스크 포인트·구현 패턴·아키텍처 선택지로 **구조화**한다.

이때 수많은 사용자의 외부 탐색이 병렬적으로 플랫폼 내부의 도메인 컨텍스트 DB로 전환된다 — 일종의 **Human Leverage**. 각자 바깥에서 발견한 정보·문제의식이 유입되고, 플랫폼은 이를 다음 사용자의 AI 에이전트가 더 빠르고 정확하게 작업할 컨텍스트로 재사용한다.

**왜 도메인 특화가 이 해자를 강화하는가**: 범용 앱 생성 플랫폼은 사용자가 만들려는 것(예약앱·CRM·쇼핑몰·게임…)이 너무 넓어 컨텍스트가 흩어진다. 반면 금융/주식/전략/리스크/포트폴리오로 좁힌 이 플랫폼은 들어오는 컨텍스트가 서로 연결된다(골든크로스·세븐스플릿·배당·리밸런싱·MDD 관리·섹터 로테이션·유튜버별 전략 패턴 등). 축적되면 단순 데이터가 아니라 **도메인 지식 지도**가 된다. 3.3의 다층 검증과 결합되어, 이 컨텍스트에는 성과뿐 아니라 의도·구현·근거·재현 기록이 함께 붙는다.

> 동작 루프의 상세는 5.3 Context Flywheel 참조.

### 3.6 검증 가능한 생성 — 포스트 코딩 시대의 엔지니어링 자산

AI가 코드를 생성하는 능력 자체는 빠르게 범용화된다. 플랫폼의 핵심 엔지니어링 자산은 코드를 많이 생성하는 능력이 아니라, 사용자의 모호한 의도를 실행 가능한 계약으로 변환하고 결과가 무엇을 의미하며 왜 신뢰할 수 있는지를 증명하는 구조다.

플랫폼에서 AI 생성의 결과물은 코드 자체가 아니라 다음 계보를 가진 **버전형 전략·지식 자산**이다.

```
사용자 원문·출처
  → 해석된 명세
  → 적용한 가정
  → 데이터·시간대·체결 계약
  → 기대 동작과 검증 방법
  → 실제 실행 결과
  → 실패 원인과 수정 내용
  → 사용자 의도 확인
  → 검증 상태가 붙은 버전
```

사용자가 생성한 전략과 검증 로직은 읽고 내보내며 다른 LLM으로 검토할 수 있어야 한다. 이를 위해 코드만 제공하는 것이 아니라 명세, 데이터 계약, 실행 환경, 검증 결과와 버전 계보를 함께 제공한다.

이 계보는 유사 전략 검색·추천·자동 테스트 생성·지식 교차 검증·도메인 임베딩 학습에 사용되는 고품질 데이터가 된다.

**초기 축적 원칙**: 코드는 실제 실행 동작의 정본이고, 버전 관리되는 기계가독 명세는 코드만으로 복원할 수 없는 사용자 의도·가정·실행 계약·검증 증거의 정본이다. 둘은 같은 내용을 복제하는 사본이 아니라 서로 다른 책임을 가진다. 전략은 자산을 묶는 단위이며, 실제 축적의 기본 단위는 사용자 요청·수정·검증 episode와 명세 revision이다. 각 레코드는 안정적인 ID와 provenance를 가져 어떤 요청이 어떤 명세·구현·검증으로 이어졌는지 연결한다.

초기에는 YAML 파일과 Git으로 이 모델을 실제 전략 개발에 적용해 스키마를 검증한다. 임베딩은 별도 정본이 아니라 이 구조화 자산에서 생성하는 파생 인덱스이며, 모델 학습은 데이터 규모·품질·활용 권리가 충분해진 뒤 별도로 판단한다.

## 4. 단계별 로드맵 (Roadmap)

### 초기 시장 진입 — B2C 주식 바이브 코딩

초기 사용자는 금융 지식 검증 플랫폼이라는 장기 비전보다, 자연어로 자신의 주식 아이디어를 빠르게 구현하고 백테스트할 수 있다는 직접적인 효용을 통해 유입된다.

플랫폼은 초기에는 **B2C 주식 바이브 코딩**이라는 이해하기 쉬운 진입점을 사용한다. 사용자가 유튜브·커뮤니티·책·외부 AI·자신의 아이디어에서 얻은 전략을 코딩 없이 구현하고 직접 실행·검증하는 경험을 우선 제공한다.

- **초기 유입**: 자연어로 주식 전략 구현
- **활성화**: 빠른 백테스트와 동작 확인
- **재방문**: 전략 수정·검증·지식 축적
- **장기 가치**: 금융 지식과 전략을 비판적으로 검증하는 능력
- **해자**: 검증된 전략·지식·검증 방법과 수정 계보

빠른 구현은 사용자 유입 수단이며, 검증·수정·지식 축적은 지속 사용과 장기적 해자를 만드는 제품 본체다.

### 1단계 — 레포를 제품 수준으로

- 현재 진행 중인 리팩토링(#242 등)의 종착점
- 기준: **사용자가 쓸 때 내부 아키텍처 관련 사항이 절대 새어나오면 안 됨**
- 내부 정리는 출시의 필수 전제 조건

### 2단계 — 웹 서비스화

- 브라우저에서 사용하는 트레이딩/리스크 분석 서비스로 공개
- 초기 목표는 **자동매매가 아님** — 백테스트, 리스크 분석, 전략 체험 중심

### 3단계 — 외부 AI의 기반(Substrate)화

- 사용자가 자기 AI(ChatGPT, Claude 등)에게 물으면, 그 AI가 **플랫폼이 제공하는 아키텍처를 참조**해 사용자만의 확장을 구축
- 플랫폼 = 완성품 앱이 아니라 **외부 AI들이 딛고 설 수 있는 기반**

### 4단계 — 자생적 생태계 & 플라이휠

- 확장 과정에서 나오는 문제(아키텍처적/기타)가 전부 서비스 내부로 피드백 → 강화
- 사용자 유입 증가 → 사용자들의 자체 아키텍처 구축·공유
- 학습 DB 축적 → 시간이 갈수록 더 다양한 요구에 더 높은 퀄리티로 대응

## 5. 피드백 플라이휠 설계 (4단계 상세)

### 5.1 데이터 캡처 메커니즘 (두 축)

**메커니즘 1 — 인앱 AI 대화 (메인 경로)**

- 플랫폼 내에서 AI API(예: Claude API)로 사용자가 직접 대화
- 멤버십 모델: 실제 API 가격보다 약간 저렴하게 제공 ↔ 대화 데이터가 사업 자산
- ⚠️ 리스크: 파워유저 마진 역전 → 사용량 상한/티어 설계 필수
- ⚠️ 금융 대화 민감성 → 약관 명시적 동의 + 옵트인 인센티브 구조 필요

**메커니즘 2 — 요청 = 캡처 지점 (안전망, 초크포인트 설계)**

- 대화가 서비스 밖(외부 AI)에서 일어나도, **실행하려면 요청은 결국 서비스 안으로 들어와야 함**
- 요청 자체가 자연스러운 데이터 캡처 지점
- 요청은 잠재적 데이터가 되며, 사용자 승인·실행·검증·수정 기록이 연결될 때 양질의 자산으로 전환됨
- "컨텍스트를 많이 줄수록 결과가 좋아진다"를 사용자가 체감하면 대화 전체 공유도 자발적으로 발생
- → 이렇게 유입된 컨텍스트가 도메인 자산으로 전환되는 전체 구조는 **5.3 Context Flywheel** 참조

**설계 함의**: 요청 포맷 자체를 풍부하게 — 단순 명령이 아니라 의도·근거·참고 지식을 담는 **구조화된 요청** 형태로 받으면 검증 가능한 데이터로 전환할 가능성이 높아짐. 최종 품질은 이후의 사용자 승인·실행·검증·수정 기록으로 판별함.

### 5.2 축적되는 데이터

플랫폼은 단순 대화 로그나 백테스트 수익률이 아니라 다음 데이터를 연결해 저장한다.

1. **의도 데이터**: 사용자 원문, 영상·문서 출처, 추출된 주장, 사용자 승인 명세
2. **실행 계약 데이터**: 데이터 주기, 타임존, 거래 세션, 체결 모델, 비용과 예외 조건
3. **검증 데이터**: 기대 동작, 자동 테스트, 실제 주문·체결, E2E 결과, 반례와 재현 기록
4. **수정 데이터**: 실패 원인, 수정 전후 차이, 사용자 확인, 버전 계보
5. **성과 데이터**: 수익률, MDD, 샤프, 표본외 결과와 스트레스 테스트
6. **지식 관계 데이터**: 서로 지지·충돌·보완하는 주장, 과거와 현재 지식의 변화, 새롭게 발견된 관계
7. **제품 효용 데이터**: 의도 확정까지 필요한 질문 수, 최초 명세 승인율, 승인 후 재수정률, 실행·검증까지 걸린 시간, 중단 지점, 재사용률과 사용자 만족도

초기 파일럿에서는 전략별 디렉터리 아래 요청 episode와 명세 revision, 구현 참조, 검증 증거를 ID로 연결한다. 검증은 단일 `verified` 값이 아니라 의도·구현·데이터 계약·타임존·체결·통계·재현 등의 독립된 상태와 적용 범위를 기록한다. 에이전트는 공통 검증 항목과 전략별 추가 항목을 제안하고 자동 실행할 수 있지만, 사용자 의도와 가정을 임의로 확정하지 않는다. 소급 기록은 당시 확인한 사실과 사후 복원·추정·미확인을 구분한다.

사람이 읽는 기대 동작 중 자동화 가능한 사례는 기계가독 입력과 예상 결과로 저장하고 CI에서 실제 전략 실행 결과와 비교한다. 검색은 데이터 규모에 맞춰 `초기: 에이전트가 YAML 직접 탐색 → 중기: yq/Python 구조 검색 → 장기: DB·구조 검색·임베딩 인덱스`로 확장한다. 저장 기술을 미리 확정하기보다 의미와 관계를 잃지 않는 레코드 형식을 먼저 검증한다.

특히 실패와 수정 기록은 단순 오류 로그가 아니다. 다음 사용자의 유사한 요청에서 확인해야 할 가정, 추천할 검증 항목, 피해야 할 구현 패턴을 알려주는 핵심 학습 자산이다.

플랫폼은 생성량이나 질문 수 감소만을 성공으로 보지 않는다. 목표는 **필요한 최소한의 질문으로 사용자의 의도를 정확히 파악하고, 구현과 검증에 걸리는 시간 및 불필요한 재작업을 줄이는 것**이다.

### 5.3 Context Flywheel — 외부 금융 지식의 플랫폼 자산화

수많은 사용자가 각자 외부에서 만든 금융 컨텍스트가 플랫폼으로 유입되어 도메인 자산으로 축적되는 루프. (해자 관점의 정의는 3.5 참조, 여기서는 동작 방식을 기술.)

**루프**:

```
외부 지식/사용자 대화/전략 아이디어
  → 주장과 의도 추출
  → 전략·지식 명세와 가정 구조화
  → 유사한 검증 자산 검색
  → 구현·실행·검증
  → 실패·반례·수정·사용자 승인 기록
  → 검증 상태가 붙은 버전형 자산 축적
  → 다음 사용자의 의도 파악·추천·구현·검증 품질 향상
  → 더 다양한 사용자와 외부 지식·컨텍스트 유입
```

**루프가 성숙하며 기대되는 효과**: 사용자 의도 파악 속도 향상, 전략 유형 분류 정확도 향상, 적절한 아키텍처 선택 확률 향상, 구현 템플릿 추천 정확도 향상, 검증 항목 자동 추천, 자주 등장하는 틈새 니즈 발견, 커뮤니티·증권사·유튜브에서 반복되는 관심사 지도화, 도메인 특화 AI 에이전트 품질 향상.

**한 줄 정의**: 사용자는 외부 금융 지식을 가져오고, 플랫폼은 그것을 실행 가능한 컨텍스트로 바꾸며, 그 누적 컨텍스트가 다음 사용자의 AI를 더 똑똑하게 만든다.

### 5.4 검증된 자산과 비강제 추천

사용자가 전략이나 금융 지식을 제시하면 플랫폼은 임베딩으로 유사 후보를 찾고, 구조화된 조건과 검증 범위를 다시 비교한다.

일정 검증 수준과 적용 조건을 충족한 후보만 대화 중 **“혹시 이것을 찾으셨나요?”** 형태로 제안한다. 추천에는 공통점, 차이점, 검증 상태, 적용 시장과 알려진 한계를 함께 표시한다.

사용자는 추천을 선택하거나 무시하고 자신의 아이디어로 계속 진행할 수 있다. 유사 후보가 없으면 억지로 연결하지 않고 신규 자산으로 시작한다.

추천은 다음을 구분한다.

- **검증된 유사 자산**: 실제 대체 후보가 될 수 있는 높은 일치도의 전략·지식
- **연관된 관점**: 직접 대체할 수는 없지만 새로운 관계와 관점을 제공하는 탐색 후보
- **검증 방법 템플릿**: 해당 주장이나 전략을 어떤 방식으로 확인할지 제안하는 검증 자산

임베딩 유사도와 인기도는 후보 발견에만 사용하며 동일성·정확성·신뢰성을 단독으로 결정하지 않는다.

자산의 성숙 단계는 `Draft → Interpreted → Confirmed → Executable → Official`처럼 표현할 수 있다. 다만 검증은 하나의 선형 단계가 아니라 의도 일치도, 구현 검증도, 데이터·체결 계약, 통계적 유효성, 재현 가능성과 출처 추적성을 독립적으로 표시한다. 검증 결과는 특정 명세·코드·데이터·조건에 연결하며, 그 범위 밖의 신뢰를 의미하지 않는다.

## 6. 검증 레이어 (핵심 차별점)

### 6.1 다층 검증

플랫폼은 검증을 하나의 점수나 백테스트 결과로 축소하지 않는다.

| 검증 층 | 핵심 질문 | 주요 수단 |
|---|---|---|
| 의도 검증 | 사용자가 원한 내용을 정확히 해석했는가 | 원문 대조, 명세 확인, 사용자 승인 |
| 구현 검증 | 명세대로 실행되는가 | 자동 테스트, E2E, 이벤트·상태 관찰 |
| 지식 검증 | 주장과 근거가 사실적·논리적으로 타당한가 | 출처 확인, 교차 검증, 반례 탐색 |
| 통계 검증 | 관측된 성과가 유효한가 | 표본외 검증, 민감도 분석, 편향·누수 검사 |
| 재현 검증 | 다른 조건에서도 결과를 확인할 수 있는가 | 독립 재실행, 데이터·기간·시장 변경 |
| 적용 검증 | 특정 사용자와 상황에 적용 가능한가 | 가정·제약·위험 조건 비교 |

각 검증 결과는 해당 전략이나 지식 버전에 연결되며, 검증 범위 밖의 내용을 보장하지 않는다.

### 6.2 지식 검증 레이어 상세

- 사용자가 유튜브·책·커뮤니티에서 본 내용, 그리고 **자신의 AI가 생성한 전략·분석**을 가져와 "공부해보자 / 포트폴리오에 참고해보자" 할 때 — 특히 3단계(substrate) 이후에는 **사용자 AI의 출력 자체가 검증 대상 1순위**가 된다:
  - 정보의 **수학적 오류·사실 오류를 탐지** (필요 시 논문 인용)
  - **비판적 피드백** 제공
- **통계 축적 효과**: 많은 사람이 공통적으로 가져오는 전략(예: 골든크로스, 배당주 몰빵)은 공통 약점이 있음 → 같은 유형 요청이 쌓일수록 약점 데이터도 쌓임 → 시간이 갈수록 사용자 의도를 더 정확히 파악하고 약점을 더 정확하게 짚어줄 수 있게 됨 (이 축적은 3.5·5.3의 Context Flywheel과 같은 엔진에서 나온다 — 지식 검증 정확도 역시 컨텍스트 누적의 수혜를 받는다)
- 특화된 검증 에이전트들이 사용자가 가져온 정보의 합당성 분석 + "이 전략을 계좌에 적용하면 말이 되는가" 수준의 판단 보조

### 6.3 검증 도구의 생태계화

- 측정/검증 도구(플러그인) 자체도 생태계 대상:
  - **공식 트랙**: 인증된 주체가 검증한 파인튜닝 모델 등
  - **퍼블릭 트랙**: 누구나 등록
- 퍼블릭 도구는 **백그라운드 리뷰어 AI의 사전 점검**과 독립 재현·실패·수정 기록을 통해 검증 상태를 축적한다. 사용량·스타·포크 같은 커뮤니티 신호는 발견과 채택의 보조 지표로만 사용한다.

**설계 원칙 — 다수 합의 ≠ 정확성**: 금융은 군중 합의가 체계적으로 틀릴 수 있는 도메인(밈 주식 등). 사용자 데이터 기반 파인튜닝·교차 검증의 신뢰도 산식은 출처·실행 결과·독립 재현·반례 등 검증 축에 앵커링하며, 사용량·합의 신호와 구분한다. "많이 들어온 전략 = 약점 데이터가 쌓임"이지 "많이 들어온 전략 = 신뢰 상승"이 아니다.

### 6.4 왜 검증이 해자인가

- "전략 공유 커뮤니티"는 트레이딩뷰가 이미 존재, "AI 전략 생성"은 곧 범용화
- **"이게 헛소리인지, 네 상황에서 말이 되는지"를 시스템적으로 판정**하는 것은 아무도 제대로 못 하고 있음
- risk_desk의 Evaluator 패턴 = 검증 에이전트의 원형, 백테스트 엔진 = 실행 결과를 정량적으로 관찰하는 도구 → 프로젝트 DNA와 정확히 일치
- 검증이 어렵기 때문에 해자가 됨

### 6.5 사용자 설계형 검증 방법과 협업

사용자마다 지식과 전략의 성격에 따라 필요한 검증 방식이 다르다. 플랫폼은 특정 검증 화면을 핵심 방식으로 강제하지 않는다.

차트 신호 시각화, 주문·체결 추적, 기간별 민감도 분석, 반례 탐색, 출처 교차 검증, 논리 검토, 다른 데이터에서의 재현 등은 선택 가능한 검증 방법의 예시다.

사용자는 검증 방법을 선택·조합하거나 자신의 검증 파이프라인을 설계할 수 있다. 검증 방법 자체도 버전형 자산으로 관리하며 다음 계보를 제공한다.

```
문제와 생성 계기
  → 검증 가설
  → 구현 방식
  → 통과한 검증
  → 발견된 실패와 수정
  → 알려진 한계
  → 현재 적용 범위
```

일정 수준 이상 검증된 방법은 템플릿이나 옵션으로 추천한다. 사용자는 해당 방법이 만들어진 배경과 근거를 시간순으로 검토한 뒤 채택할 수 있다.

사용자는 전략과 결과뿐 아니라 검증 파이프라인을 가족·친구 등 필요한 사람과 공유할 수 있다. 공유받은 사용자는 같은 조건으로 재실행하거나 다른 기간·종목·가정으로 비교하고, 반례를 추가하거나 파이프라인을 복제·수정할 수 있다.

플랫폼의 사회적 기능은 결론을 확산하는 커뮤니티가 아니라 **검증 과정을 재현하고 서로의 비판적 사고 방식을 배우는 협업 공간**을 지향한다.

## 7. 알려진 리스크 & 열린 질문 (Open Questions)

### 7.1 검증 레이어의 함정

- **백테스트 좋음 ≠ 좋은 전략** (오버피팅, 생존편향)
  - 리뷰어 AI가 "백테스트 수익률 높음 = 신뢰도 상승"으로만 가면 오버피팅 전략이 상위 랭크되는 트레이딩뷰의 고질병을 물려받음
  - 대응 방향: 아웃오브샘플 검증, 포워드 테스트 기간 강제 등을 신뢰도 산식에 포함
  - 보완 요소: 지식 검증 레이어(6.2)는 성과 숫자와 독립적이라 이 함정을 논리·근거 레벨에서 보완

- **집단 검증 ≠ 집단의 정확성**
  - 조회수·좋아요·댓글 수는 관심도의 신호이지 정확성의 증거가 아니다.
  - 인기 의견을 우선 노출하면 사용자의 판단이 다수 의견에 종속되고 잘못된 믿음이 반복 강화될 수 있다.
  - 인기보다 출처 추적성, 독립 재현, 반례 기록, 검증 범위와 적용 조건을 우선한다.
  - 사용자 평가는 의도 일치와 효용 신호로 활용하되 사실적·통계적 정확성 라벨과 분리한다.
  - 다른 사람의 결론을 보기 전에 자신의 판단을 기록하는 독립 검증 방식도 고려한다.

- **프런티어 모델 ≠ 진실 판정기**
  - 더 발전한 모델도 사실 오류, 시대착오, 출처 오해와 과도한 확신을 만들 수 있다.
  - 모델 간 합의도 독립된 사실 증거로 취급하지 않는다.
  - 모델은 주장 추출, 관계 탐색, 반례 제안과 검증 보조에 사용하고 최종 근거는 출처·데이터·재현 기록에 연결한다.

### 7.2 규제 경계

- "네 계좌에 이 전략 써도 됨/안 됨" 판정은 **투자자문업** 경계와 흐려질 수 있음 (한국 규제 환경에서 중요)
- 안전한 포지셔닝: **교육/리터러시/시뮬레이션 도구** — "가져온 정보가 맞는지, 알려진 한계가 뭔지 알려주는 것"
- 계좌 연동 판정까지 가는 것과의 법적 선 확인 필요 (미해결)

## 8. 포지셔닝 요약

- **초기 카테고리**: 자연어로 주식 아이디어를 구현하고 백테스트하는 B2C 주식 바이브 코딩 플랫폼
- **사용자 약속**: 금융 아이디어를 필요한 최소한의 상호작용으로 정확히 구조화하고, 검증 가능한 소프트웨어와 증거로 빠르게 변환
- **장기 정체성**: 금융 지식·전략을 구현·관찰·검증·수정하며 비판적 판단력을 기르는 플랫폼
- **핵심 성공 요인**: 금융 도메인 의도 구현 품질 × 검증 가능성 × 개인 효용
- **장기 해자**: 검증된 전략·지식·검증 방법, 실패·수정 계보, 다양한 B2C 사용자가 만든 재현 가능한 긴 꼬리 데이터
- **아님**: 자동매매 툴, 개발자용 툴킷, 플랫폼 내부 전체 공개가 목적인 오픈소스 프로젝트, 인기순 금융 콘텐츠 피드, 좋아요 기반 전략 랭킹, 다수 의견을 정답으로 취급하는 투자 커뮤니티
- **맞음**: 검증된 기반 위에서 높은 자유도를 제공하고, 사용자가 외부 정보와 자신의 아이디어를 직접 구현·검증할 수 있게 하는 금융 리터러시 플랫폼

빠른 구현은 사용자를 들어오게 하고, 검증 경험은 다시 사용하게 하며, 축적된 검증 자산은 다음 사용자의 구현과 판단을 개선한다.

우리는 금융 결론의 확산을 최적화하지 않는다. 사용자가 주장과 근거를 분리하고, 검증 과정을 설계·관찰·재현·공유하면서 스스로 판단하는 능력을 기르도록 한다.
