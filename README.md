# context-log

비개발 컨텍스트 로그 — 고민, 아이디어, 사업, 리서치, 의사결정의 **과정**을 기록하는 저장소.
코드는 없고, 이슈 트래커가 본체다.

## 운영 원칙

- **이슈 = 컨텍스트의 원자 단위.** 1이슈 1주제.
- **결과가 아니라 과정을 보존한다.** 당시의 고민, 검토한 선택지, 기각한 이유가 핵심 자산.
- **open 이슈는 두 종류뿐:** `backlog`(초기 아이디어/나중에 볼 것) 또는 `debating`(진행 중인 고민).
- **closed = 폐기가 아니라 확정된 기록.** 결론이 난 것, 기록만 남기는 것은 close한다.
- 후속 생각은 새 이슈가 아니라 **기존 이슈에 코멘트**로 추가한다 (사고 흐름을 한 스레드에 시간순 보존).

## 라벨 체계

> 라벨 체계는 추후 필요에 따라 변경될 수 있다. 변경 시 이 README를 갱신한다.

**상태 라벨** — 이슈당 정확히 1개, open/closed 상태를 결정:

| 라벨 | 의미 |
|---|---|
| `backlog` | 초기 아이디어, 초기 고민, 나중에 할 일. 숙성 대기 중 |
| `debating` | 어느 정도 쌓였거나 진행 중인 고민. 활발히 검토 중 |

**주제 라벨** — 이슈당 1개 이상:

| 라벨 | 의미 |
|---|---|
| `idea` | 제품/서비스/기능 아이디어 |
| `business` | 창업, 사업 전략, 수익화 |
| `research` | 시장/기술/경쟁 리서치 |
| `feedback` | 외부에서 받은 피드백 |
| `decision` | 확정된 의사결정 (결론 도출 시 부착) |

## 라이프사이클

```
초기 아이디어/고민 ──▶ [backlog] open
                         │ 숙성/착수
                         ▼
진행 중인 고민   ──▶ [debating] open ── 코멘트로 검토 과정 축적
                         │
             ┌───────────┴───────────┐
             ▼                       ▼
    결론 도출: decision 라벨    기각: 이유를 마지막
    추가 + 결론 코멘트 + close   코멘트로 남기고 close
```

이미 결론이 난 결정, 받은 피드백, 리서치 메모 등 "진행"이 없는 기록형 이슈는 생성 즉시 close한다.

## 작성 규칙

모든 작성자(AI 에이전트 포함)는 [AGENTS.md](./AGENTS.md)를 따른다.

## 하위 논점 분리 규칙

큰 논의에서 독립적으로 추적할 만큼 커진 논점은 별도 이슈로 분리할 수 있다.

GitHub의 sub-issue 기능을 직접 쓰지 않고 `References`로 연결하는 경우:

- 자식 이슈 제목에는 부모 이슈 번호를 표시한다.
  - 형식: `[주제라벨] #부모번호 하위 — 핵심 주제`
  - 예: `[business] #3 하위 — 도메인 컨텍스트 플라이휠 / Human Leverage`
- 부모 이슈 본문에는 `분리된 하위 논점` 목록을 짧게 추가한다.
  - 예: `#4 — 도메인 컨텍스트 플라이휠 / Human Leverage`
- 자식 이슈 `## References`에는 부모 이슈를 연결한다.

## 관련 저장소

- 개발 컨텍스트: 각 프로젝트 레포의 이슈 (예: `open-trading-api`)
- 리치 콘텐츠(이미지, 임베드, PDF)가 필요한 리서치: Notion 페이지를 만들고 이슈 `## References`에 URL 연결
